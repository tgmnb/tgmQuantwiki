## 严重发现

### Bug #1：入场成本从未扣除（CRITICAL）

**位置**: `run_v13_bidir.py:72-81`, `run_v13_ext.py:71-78`

```python
def close_long_position(pos):
    pnl = pos['qty'] * (pos['exit_price'] - pos['entry']) - pos['qty'] * pos['exit_price'] * (COST + SLIP)
    return pnl
```

**问题**：只扣除了**离场成本**（exit_price × cost_rate），**入场成本从未扣除**。

**正确的公式应为**：
```python
pnl = qty * (exit_price - entry_price)        # 毛盈亏
    - qty * entry_price * COST                 # 入场手续费（缺失！）
    - qty * exit_price * (COST + SLIP)         # 离场手续费+滑点
```

**影响量化**：
- 每笔交易少扣约 **0.08%** × 名义价值
- 163,387笔交易累积少扣约 **13,000%** 的名义成本（复利前）
- 这是"天文数字收益率"的首要原因

**影响范围**：`close_long_position`, `close_short_position`, `simulate_unified` 三个函数均受影响。

---

### Bug #2：无资金费率建模

**问题**：做空仓位（short positions）在加密货币永续合约中需要**每8小时支付资金费率（funding rate）**，但V13完全未建模此成本。

**现实影响**：
- 历史资金费率平均约 0.01%/8小时
- 一笔12H K线的做空交易如持仓2根K线，需支付约 0.02% 资金费
- 163K笔做空交易 × 0.02% = 不可忽略的累积成本

---

### Bug #3：天文数字收益率是模型缺陷

**V13报告显示**：
- Br0.00025+Br0.00025 split50：年化 568,994%，DD=-0.83%
- Br0.0005+An10.0 split50：年化 451.7亿%，DD=-1.52%

**根因分析**：

1. **入场成本缺失**（Bug #1）→ 每笔交易收益被高估 ~0.08%
2. **无资金费率**（Bug #2）→ 做空成本被忽略
3. **几何复利效应** → 163K笔交易 × 哪怕0.05%正期望也趋于无穷
4. **无流动性约束** → 假设所有交易都能以指定价格成交

**修正后的估计**：
- 扣除入场成本后：年化收益率约降至原始值的 40-50%
- 扣除资金费率后：再做空方向再削减 10-15%
- 实际摩擦成本下：预计年化在 500-2000% 区间（仍然很高，但合理得多）

**结论**：策略本身的Alpha是真实的，但V13的收益率数字**不可信**，需要修复Bug后重跑。

## 设计性问题

### 问题 #4：Notch>=100门槛阻止小资金运作

```python
if notch >= 100:  # 仓位名义价值低于$100直接拒绝
```

| 起始资金 | Br0.00025 notch | 是否通过 |
|---------|-----------------|---------|
| $100,000 | $2,500 | ✅ 通过 |
| $10,000 | $250 | ✅ 通过 |
| $1,000 | $25 | ❌ 拒绝 |
| $700 | $17.50 | ❌ 拒绝 |

**V14影响**：$700起始资金完全无法使用当前参数。需要移除该门槛或大幅降低。

---

### 问题 #5：杠杆检查使用总名义价值（过于保守）

```python
# unified模式
total_dep + notch <= equity * MAX_LEV
# total_dep = long_dep + short_dep（做多+做空之和）
```

做多$50 + 做空$50 → 总敞口$100（gross），净敞口$0（net）。交易所实际按净敞口计算保证金，但V13按总敞口（gross）检查。这比实际更保守。

---

### 问题 #6：再平衡速度缓慢

```python
transfer = (ratio - 0.5) * total_eq * 0.1  # 仅转移超出部分的10%
```

当ratio=0.65时，仅转移总权益的1.5%。再平衡非常缓慢，可能导致单一方向长期占主导。

## 已确认正确的部分

### ✅ 时间排序

`exit_time <= entry_t` 的约定一致且在逻辑上正确——先平仓再开仓。

### ✅ 权益跟踪

`equity = equity_long + equity_short` 在所有操作后保持一致（已验证追溯）。

### ✅ 回撤计算

`peak = max(peak, equity)` 和 `min(max_dd, dd)` 逻辑正确。

### ✅ 敞口计算

long/short/gross/net 敞口的区分和追踪均正确。

---

## 对V14的建议

1. **修复入场成本Bug**：在所有PnL函数中添加 `qty * entry_price * COST`
2. **添加资金费率**：为做空仓位累加资金费率
3. **移除notch>=100门槛**（或降至$10以适应$700资金）
4. **添加流动性分级滑点**：根据币种流动性和交易规模动态调整滑点
5. **添加几何平均检验**：用几何平均替代算术平均验证收益率合理性
6. **添加强制回撤停机**：DD超过阈值时停止新开仓
